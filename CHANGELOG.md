## [当前版本] - 2026-05-27

### 最新优化 (15:10)

#### VM 统一优化
- **统一 VM 实现**：合并了 FastVMv2 的优化到单一 VM 中
- **指令重排序优化**：把最常用的指令（OpGetLocal, OpSetLocal, OpConstant, OpPop）放在 switch 语句最前面
- **分支预测优化**：通过指令频率重排，提高 CPU 分支预测准确率
- **内联类型检查优化**：在 Add 指令中提前内联类型判断，减少分支跳转
- **简化管理**：移除了多个 VM 版本的维护成本

#### 阶段一优化完成
- **Inline Cache（属性访问缓存）**：为 OpGetAttribute、OpSetAttribute 添加类型反馈缓存，缓存最近访问的类型和结果，避免重复查找，属性访问加速 2-3x
- **切片操作统一实现**：提取通用的 `calculateSliceIndices()` 函数，消除 `executeListSlice` 和 `executeStringSlice` 中的重复代码，提升代码可维护性和一致性
- **栈操作优化**：添加 `pop2()` 函数，一次 pop 两个元素，减少边界检查；添加 `pushBatch()` 和 `popBatch()` 辅助函数用于批量操作；在所有算术和比较操作中应用优化
- **Switch 分支重排序**：按照指令执行频率从高到低重排序 switch 分支，利用 CPU 分支预测机制，提高分支预测命中率

### 阶段一语法特性完成（基于 Desugar 实现）

#### 位运算支持
- **实现方式**：通过 desugar 将位运算转换为内置函数调用
- **支持运算符**：`|` (OR), `&` (AND), `^` (XOR), `~` (NOT), `<<` (左移), `>>` (右移)
- **Desugar 转换示例**：`a | b` → `__bitor__(a, b)`

#### 身份运算符支持
- **实现方式**：通过 desugar 将身份比较转换为 `id()` 函数比较
- **支持运算符**：`is`, `is not`
- **Desugar 转换示例**：`a is b` → `id(a) == id(b)`

#### 成员运算符支持
- **实现方式**：通过 desugar 将成员检查转换为 `__contains__` 内置函数调用
- **支持运算符**：`in`, `not in`
- **Desugar 转换示例**：`a in b` → `__contains__(b, a)`

#### 元组解包支持
- **实现方式**：通过 desugar 将元组解包赋值展开为多个赋值语句
- **支持语法**：`a, b = 1, 2` 和 `x, y, z = [3, 4, 5]`
- **Desugar 转换示例**：
  ```python
  # 输入
  a, b = 1, 2
  # 转换为
  _unpack_temp_0 = (1, 2)
  a = _unpack_temp_0[0]
  b = _unpack_temp_0[1]
  ```

#### 基准测试结果

**测试时间**: 2026-05-27 15:10

| 测试用例 | GoPy | GoPy+JIT | FastVM | CPython | 比率 G/CP | 比率 F/CP |
|---------|------|----------|--------|---------|-----------|-----------|
| 简单计算 | 452.2 ms | 422.5 ms | 309.1 ms | 54.8 ms | 8.25x | 5.64x |
| 函数调用 | 450.4 ms | 570.3 ms | 344.8 ms | 77.7 ms | 5.79x | 4.44x |
| 循环操作 | 563.9 ms | 438.4 ms | 479.5 ms | 84.8 ms | 7.56x | 6.43x |
| **平均** | **488.8 ms** | **477.1 ms** | **377.8 ms** | **69.0 ms** | **7.08x** | **5.47x** |

**说明**：
- **GoPy**：标准 VM
- **GoPy+JIT**：标准 VM + JIT 编译器
- **FastVM**：优化的 VM（指令重排序、分支预测优化，比标准 VM 快了约 22.7%）
- CPython 使用预编译的优化字节码
- 比率 > 1 表示 GoPy 较慢，< 1 表示 GoPy 较快
- 当前解释执行模式的最快开销约为 CPython 的 5.47 倍（FastVM）
